# uv

## Intro

UV is a modern, high-performance Python package manager written in Rust, designed as a fast alternative to pip or conda that simplifies dependency management independently of Python versions.

## Installation

Homebrew

``` numberSource
brew install uv
```

Pip

``` numberSource
pip install uv
```

## Initialization

If you don’t have a project directory run the following command:

``` numberSource
uv init <project-title>
```

If you already have a project directory `cd` into the directory and run the following command:

``` numberSource
uv init
```

When initializing the project, there are a few files that will be populated: `.python-version`, `hello.py`, `pyproject.toml`, and `uv.lock`.

Let’s go over what each one does:

- `.python-version`: this is the file used to manage the python version. For example, let’s say that it defaults to python version 3.13 because it’s your global python’s version but you want to use a different version, then you can modify it to 3.12.
- `hello.py`: this is just a basic Python template, so it doesn’t carry much meaning on its own and can safely be deleted.
- `pyproject.toml`: this file is the core of setting up your coding environment. It will be explained further below, but it manages the Python version and can also take on the role of `requirements.txt`.
- `uv.lock`: there’s no need to touch this file directly.

## Virtual Environment

### Creating the virtual environment

After initializing the project, running the command below will automatically do what `python3 -m venv .venv` used to do when creating a virtual environment manually:

``` numberSource
uv sync
```

This creates the `.venv` virtual environment and installs the dependencies by referencing the `.python-version`, `pyproject.toml`, and `uv.lock` files. In fact, you don’t even have to run it explicitly — running any Python script with `uv run <script.py>` will create the virtual environment the same way if it doesn’t exist yet.

### Activating and deactivating

> **NOTE:**
>
> Use `uv run` for any scripts or packages as uv auto detects your virtual environment.

Normally, every time you come back to a project you would have to activate the virtual environment before running anything:

``` numberSource
source .venv/bin/activate
```

You would then see the terminal prompt change from `username project-directory` to `(.venv) username project-directory`, and when you’re done, deactivate it:

``` numberSource
deactivate
```

With uv, none of this is necessary. `uv run` automatically detects the `.venv` in your project and runs your script inside it, so you never have to remember to activate the environment every time you return to the project. The commands above are only there in case you want to work inside an activated shell manually.

## Package Management

This part is where uv truly shines. You no longer have to sit through long download times for large libraries or decipher cryptic error messages from version conflicts like you would with pip or conda.

### Adding and removing packages

``` numberSource
# Add a package
uv add <package-name==version>

# Remove a package
uv remove <package-name==version>
```

Occasionally a package name comes with extra options in square brackets. In that case, handle it as follows:

``` numberSource
# For example
pip install "fastapi[standard]"

# The uv way
uv add fastapi --extra standard
```

Think of `--extra` as replacing the double quotes and square brackets.

### Exporting dependencies

Packages are automatically recorded in `pyproject.toml` along with their versions, but if you’re new to uv and find `requirements.txt` more comfortable, you can export them by running the following commands in the terminal:

``` numberSource
uv pip freeze > requirements.txt

# To install packages from requirements.txt
uv pip install -r requirements.txt
```
